## What went well

* Clean Cappy design system components (`Field`, `Button`, `RowItem`, `Sheet`, `Card`, `Avatar`) made laying out pages fast, visually consistent, and highly polished.
* The TypeScript compiler was very helpful in identifying missing API methods (`getChild`) and mapping types early.

## What was harder than expected

* Exposing the typed `supabase` client with the explicit `: SupabaseClient<Database>` annotation was overriding generic default parameters and mapping all table inserts to `never[]`. Letting type inference resolve it directly from `createClient<Database>()` successfully fixed the typechecker errors.
* The Deno Edge Functions in the `supabase/functions` directory were being picked up by React Native's `tsconfig.json` compiler, causing Deno runtime import errors. Excluding this directory resolved the issue cleanly.
* React Native style arrays caused compatibility errors on custom components because design system properties used `ViewStyle` instead of `StyleProp<ViewStyle>`.

## What I had to invent or assume

* **Weight conversions:** App user inputs weight in pounds (`lb`), but database schema requires grams. Implemented a multiplier conversion `Math.round(weightLb * 453.59237)` on submit.
* **Premium iOS Date Picker:** Replaced standard iOS system alerts with the design-system `Sheet` component to host the Wheel/Spinner `DateTimePicker`, resulting in a significantly more premium and tailored Cappy layout.
* **History joins:** Designed detailed Postgrest joins (`listDosesWithDetailsForChild`/`listDosesWithDetailsForFamily`) so history lists can pull medication brand/generic names and caregiver display names.

## Tech debt or follow-ups created

* **Card Props (Low):** Modified the `Card` component properties in `Card.tsx` to support `StyleProp<ViewStyle>` so screens can pass array styles.

## Suggested change to my own system prompt

* When initializing Supabase clients in TypeScript, let the compiler infer the client's type directly from `createClient<Database>()` rather than annotating it explicitly with `SupabaseClient<Database>` to prevent default generic conflicts.
