## What was harder than expected

- The `owner` field in `app.json` needed to be `closedose` (the org account), not `nsm0101` (the personal account), because the EAS project ID was created under the `closedose` organization. The task spec said to use the Expo username, which was ambiguous when the user has both a personal and org account.
- `eas login` and `eas build` require interactive terminal input for credentials — these cannot be run through a non-interactive command runner. Had to hand off to the founder for those steps.

## What I had to invent or assume

- Determined that `closedose` was the correct owner by reading the error message from `eas build:configure` which stated the project ID belonged to `closedose`.

## Tech debt or follow-ups created

- **`ios.buildNumber` in app.json (Low):** EAS warned that `buildNumber` in app config is ignored when version source is set to `remote`. Should remove `"buildNumber": "1"` from `app.json` → `ios` to avoid confusion.
- **EAS CLI version (Low):** CLI reported `eas-cli@20.5.0 is now available`. Should upgrade with `npm install -g eas-cli` when convenient.

## Suggested change to my own system prompt

- When configuring EAS projects, always verify the `owner` field matches the account that owns the EAS project ID by checking `eas whoami` for available accounts, rather than assuming the personal username is correct.
