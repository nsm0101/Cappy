# Access Management Policy

> **Status:** Stub.

**Owner:** Founder
**Implements:** 45 CFR §164.308(a)(3) and §164.308(a)(4)
**Last reviewed:** [DATE]

## Principle

Least privilege. Every workforce member, system, and integration has
the minimum access required to perform its function, and no more.

## Provisioning

A new workforce member receives access through a documented checklist:
- Identity verified
- Role assigned with explicit list of permissions
- MFA enrolled
- Workstation security verified (full-disk encryption, screen lock,
  password manager)
- Training completed (`/compliance/hipaa/workforce-training.md`)
- Access added to systems
- Provisioning record retained

## Periodic review

Quarterly, the founder reviews:
- Every account on every production system
- Every API key, token, and service credential
- Every BAA-covered vendor's user list
- Audit logs for any unexpected access patterns

The review and any changes are documented.

## Termination

When a workforce member departs:
- All accounts disabled within 24 hours
- All credentials and tokens rotated
- All issued devices wiped
- Termination record retained

For alpha (single-founder), termination is theoretical until there is
a second workforce member. The procedure is documented in advance so
it is ready when needed.

## Workstation requirements

The founder's primary workstation:
- Full-disk encryption enabled (FileVault on macOS, BitLocker on
  Windows)
- Screen lock at 5 minutes of inactivity
- Strong password (24+ characters or generated by password manager)
- Auto-update enabled for OS and security patches
- No production credentials stored in plaintext on disk
- All secrets in 1Password (or equivalent), unlocked per session

## Production access

- Database access through secrets manager only; no `.env` with prod
  values on workstation
- Admin operations through audited scripts in `/infra/scripts/`
- Break-glass admin credential stored offline in a sealed envelope
  in a secure location, used only when normal access fails

## Documentation

Access changes documented in the access log (kept private; not in git).
